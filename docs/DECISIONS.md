### D-011. B2C 우선 타겟
**상태:** Accepted
**시점:** 2026-05-02
**배경:**
ARCC가 B2C(개인) / B2B(기업 직원) / B2G(군/공공) / B2H(병원/재활) 4갈래 사업 모델 가능. 한정된 자원으로 동시 추진 시 모든 채널이 미완성될 위험.
**결정:**
B2C 개인 구독을 1차 타겟으로 우선 집중. B2B/B2G/B2H는 B2C 검증 후 단계적 확장.
**근거:**

B2C는 빠른 피드백 루프 (사용자 반응 즉시 확인)
누적된 개인 코칭 히스토리 = ARCC의 차별점이자 락인 효과
본인(50대 직장인 + 하프마라톤 준비)이 1번 타겟 사용자 → dogfooding으로 품질 검증
B2H는 의료법/개인정보보호 이슈로 변호사 자문 필수 → 후순위

**검증:**
2026-09~10 정식 런칭 후 B2C 사용자 100명 도달 시점에 B2B 진출 검토.

### D-013. 인프라 단계적 확장 ($45 → $130/월)
**상태:** Accepted
**시점:** 2026-05-02
**배경:**
현재 인프라 비용 vs 정식 런칭 후 비용 격차 크면 갑작스런 부담. 사용자 증가 곡선과 비용 증가 곡선 매칭 필요.
**결정:**
2단계 인프라 로드맵 채택.

현재 (2026-05): Replit Core + Supabase Free + Anthropic API ≈ $45/월
정식 런칭 (2026-09~10): Render.com + Supabase Pro + 도메인 ≈ $130/월

**근거:**

Supabase Free는 7일 비활성 시 일시정지 → 런칭 전 Pro 업그레이드 필수
Render.com은 Replit보다 production 환경 적합
사용자 100명 시점 매출 약 99만원, 운영비 17만원 → 흑자 가능
B2C 가격 9,900원/월 가정 → 손익분기점 약 17명

**검증:**
런칭 전 Supabase Pro 마이그레이션 + Render.com 배포 테스트 완료 필요.

- 단순화 시도 = user_id 컬럼 추가 + 데이터 백필 + 정책 교체 + 회귀 테스트
- 최소 2시간 작업 + 비정규화 + 마이그레이션 위험
- 보안 효과 0 (EXISTS도 동일한 RLS 보호)
- 성능 효과 미미 (현재 데이터량 적음)

**근거:**

1. **EXISTS = 정규화된 DB 표준 RLS 패턴** — 비정상 아님
2. **5/2 정찰 시 user_id 보유 테이블만 단순화 대상으로 본 원래 판단이 합리적** — 5/6 추가 추론이 부정확
3. **보안 손실 0** — EXISTS도 동일하게 다른 사용자 데이터 차단
4. **회귀 위험 회피** — 스키마 변경 = 코드 영향 큼
5. **D-018 안전 절차** — "이유 없는 변경 금지" 정신

**검증:**

- 변경 없으므로 회귀 위험 0

**관련 메타 발견:**
